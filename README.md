# ReactiveUI Website

![Build website](https://github.com/reactiveui/website/workflows/Build%20website/badge.svg)

Source for the [ReactiveUI](https://www.reactiveui.net/) website. The site is rendered by [Zensical](https://zensical.org/) (a faster, modern-themed renderer that consumes the `mkdocs.yml` ecosystem); the API reference is generated from the published NuGet packages by [`SourceDocParser`](https://github.com/glennawatson/SourceDocParserLib) (Roslyn + ICSharpCode.Decompiler under the hood).

The build is orchestrated by [Nuke](https://nuke.build/) — it owns NuGet fetching, the API metadata walk, the explicit `nav:` generation (Zensical doesn't support awesome-pages-style auto-discovery overrides), source-link validation, and the `zensical` invocation. Python lives in a project-local `.venv/` that Nuke bootstraps on first run.

## Quick start

```bash
./build.sh BuildWebsite      # full build: NuGet fetch → API walk → nav splice → zensical build → site/
./build.sh Serve             # serve at http://127.0.0.1:8000 with hot reload
./build.sh BuildWebsite --strict   # CI-style: fail on broken links / unresolved refs
./build.sh Clean             # drop .api-cache, generated API tree, and site/
```

Direct zensical invocations work once the venv is set up (faster iteration on prose, but you have to remember to run `./build.sh WriteMkdocsConfig` after touching `mkdocs.base.yml` or any `docs/` file that should appear in the nav):

```bash
.venv/bin/zensical serve
.venv/bin/zensical build
```

Top-level configuration lives in `mkdocs.base.yml`; the actual `mkdocs.yml` zensical reads is **generated** by `Build.cs`'s `WriteMkdocsConfig` target — it splices the API tree from `SourceDocParser`'s `NavigationGraph` into the explicit `nav:` block.

## Layout

```
website/
├── mkdocs.base.yml        site config source-of-truth (theme, plugins, no nav)
├── mkdocs.yml             GENERATED by `nuke WriteMkdocsConfig`, gitignored
├── overrides/             Zensical/Material theme overrides
├── docs/                  all rendered content
│   ├── index.md           landing page
│   ├── license.md         third-party credits (Zensical, fonts, …)
│   ├── stylesheets/       brand palette + hero CSS
│   ├── docs/              handbook, getting-started, guidelines, …
│   ├── articles/          curated articles
│   ├── Announcements/     chronological announcements
│   ├── contribute/        contributor guide
│   ├── api/               GENERATED — Zensical-emitter output, gitignored
│   └── _redirects         Netlify catchall for legacy docfx .html URLs
├── build/                 Nuke build (Build.cs, NavBuilder.cs, …)
├── nuget-packages.json    package list driven by SourceDocParser.NuGet
├── requirements.txt       zensical + mkdocs-autorefs
├── .api-cache/            GENERATED — NuGet fetcher's lib/, refs/, cache/, gitignored
└── .venv/                 bootstrapped on first ./build.sh run
```

## Contributing

1. Fork and clone the repo.
2. Create a branch.
3. `./build.sh Serve` — open `http://127.0.0.1:8000` and iterate on the docs.
4. PRs against `main` build under CI (`actions/setup-python@v6` + `actions/setup-dotnet@v5`) and deploy to Netlify on merge.

For the API extraction pipeline see [`SourceDocParserLib`](https://github.com/glennawatson/SourceDocParserLib). The `SourceDocParser.Zensical` package emits the Markdown pages and exposes a `NavigationGraph` that `Build.cs` splices into the generated `mkdocs.yml`.
